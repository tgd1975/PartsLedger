## First-time setup

See [`docs/developers/DEVELOPMENT_SETUP.md`](docs/developers/DEVELOPMENT_SETUP.md)
— the canonical walk-through. It covers tool prerequisites, the install
order for Ubuntu and Windows 11, env-var configuration, and a
"common setup problems" appendix.

Short version: clone → `bash scripts/install_git_hooks.sh` →
`npm install -g markdownlint-cli2` →
`uv venv && uv pip install -r requirements-dev.txt` → `pytest`.

## Working on a task

1. Browse [`docs/developers/tasks/OVERVIEW.md`](docs/developers/tasks/OVERVIEW.md)
   for the active task list.
2. Pick a task whose prerequisites are met. Mark it active with
   `/ts-task-active TASK-NNN` before your first commit — the activation
   rides along with the commit, do not commit it separately.
3. Make your changes. Run `pytest` from repo root; ensure markdownlint is
   clean. Style conventions live in
   [`docs/developers/CODING_STANDARDS.md`](docs/developers/CODING_STANDARDS.md);
   test layout in [`docs/developers/TESTING.md`](docs/developers/TESTING.md);
   CI gates in [`docs/developers/CI_PIPELINE.md`](docs/developers/CI_PIPELINE.md);
   security-review workflow in [`docs/developers/SECURITY_REVIEW.md`](docs/developers/SECURITY_REVIEW.md).
4. Commit with `/commit "<message>" <file> [<file> …]`. The wrapper
   enforces atomic pathspec commits and rejects raw `git commit` invocations.

## Commit policy

Full rationale and the race-condition story:
[`docs/developers/COMMIT_POLICY.md`](docs/developers/COMMIT_POLICY.md).
Server-side branch protection ruleset:
[`docs/developers/BRANCH_PROTECTION_CONCEPT.md`](docs/developers/BRANCH_PROTECTION_CONCEPT.md).
The minimum rules:

- One feature / fix per commit, files listed explicitly. No `git add -A` /
  `git add .`.
- No LLM attribution trailers (`Co-Authored-By: Claude …`, "Generated with
  Claude Code", etc.) in commit messages or PR bodies.
- Pre-commit hook failures on unrelated changes follow the three-check
  protocol documented in
  [`.claude/skills/commit/SKILL.md`](.claude/skills/commit/SKILL.md)
  before any `--no-verify`.
- Bypass the security-review hook for a known-good pull with
  `PL_SKIP_SECURITY_REVIEW=1 git pull`. Logged.
- Bypass the commit-provenance hook only for interactive rebase or recovery
  with `PL_COMMIT_BYPASS="<reason>" git commit ...`. Logged to
  `.git/pl-commit-bypass.log`.

## Editing the task system

See [`docs/developers/TASK_SYSTEM.md`](docs/developers/TASK_SYSTEM.md)
for the full IDEA/EPIC/TASK workflow, lifecycle states, and the
`/ts-*` skill catalogue.

The four index files
(`docs/developers/tasks/{OVERVIEW,EPICS,KANBAN}.md` and
`docs/developers/ideas/OVERVIEW.md`) are entirely generated by
`scripts/housekeep.py`. After any task / epic / idea status change, run
`/housekeep` and sweep the regenerated indexes into the same commit as the
change that triggered them.

## Reporting issues

Open a GitHub issue. If the issue touches an existing IDEA / EPIC / TASK,
reference the identifier in the title so the cross-link is greppable.
